¿Qué construimos?

Una red social donde los usuarios publican contenido, se comentan entre sí y se agregan como amigos.

  • La fuente original de datos es un archivo Excel con 50 filas por tabla
  • El Excel se convierte a SQL y se carga en PostgreSQL al iniciar Docker
  • Desde PostgreSQL, un ETL en Python distribuye los datos a los otros tres motores
  • Corre 100% en Docker — sin dependencias locales

Restricciones CHECK

Estado solo puede ser ACEPTADA o PENDIENTE. Nadie puede agregarse a sí mismo.

CREATE TABLE amistades ( id_amistad SERIAL PRIMARY KEY, estado VARCHAR(20) CHECK (estado IN ('ACEPTADA', 'PENDIENTE')), usuario_sol_id INT, usuario_rec_id INT, CONSTRAINT chk_no_auto CHECK (usuario_sol_id <> usuario_rec_id), FOREIGN KEY (usuario_sol_id) REFERENCES usuarios ON DELETE CASCADE );
Fase 2 · PostgreSQL

Procedimientos y Vistas

Stored Procedure

CALL crear_amistad(id1, id2)

Verifica si la relación ya existe en ambas direcciones. Si existe, lanza un NOTICE en lugar de un error. Si no, inserta en estado PENDIENTE.

Tres vistas analíticas

  • solicitudes_pendientes — amistades en espera
  • amistades_consolidadas — relaciones aceptadas
  • feed_noticias — publicaciones con autor, likes y conteo de comentarios
CREATE VIEW feed_noticias AS SELECT u.nombre AS autor, p.texto_contenido, p.likes_contador, COUNT(c.id_comentario) AS total_comentarios FROM publicaciones p JOIN usuarios u ON p.autor_id = u.id_usuario LEFT JOIN comentarios c ON c.publicacion_id = p.id_publicacion GROUP BY u.nombre, p.texto_contenido, p.likes_contador ORDER BY p.fecha_publicacion DESC;

# ETL corre sin bloquear la pantalla class EtlWorker(QThread): log_line = pyqtSignal(str) finished = pyqtSignal(bool) def run(self): proc = subprocess.Popen( [sys.executable, etl_script], env={..., "PYTHONUTF8": "1"}, ) for raw in proc.stdout: self.log_line.emit( raw.decode("utf-8", errors="replace") )

Diseño

Prompt Engineering

Tres técnicas aplicadas según la complejidad de cada decisión de diseño.

Zero-Shot

SQL directo

El esquema SQL y las vistas se diseñan sin ejemplos previos. La sintaxis es estándar y la intención es clara: no hay ambigüedad en un CREATE TABLE.

Few-Shot

Documentos MongoDB

El código incluye el documento objetivo exacto antes de generalizarlo para las 50 publicaciones. El ejemplo guía la construcción del patrón.

{ "autor": {…},
  "comentarios": […] }
CTD

Schema Cassandra

Chain-Thought-Decomposition: primero la query objetivo, luego la partition key, luego el clustering key, finalmente el CQL. Cada paso justifica el siguiente.

1. Query → 2. Partition →
3. Cluster → 4. CQL

Conclusión

Cada motor hace
lo que mejor sabe.

🐘 PostgreSQL

Garantiza consistencia. Sin datos corruptos, sin amigos duplicados, sin huérfanos.

🍃 MongoDB

Un solo documento = toda la publicación. Sin JOINs para la lectura más común.

💿 Cassandra

El timeline escala horizontalmente. La partition key garantiza O(1) por usuario.

🔴 Redis

Lo más consultado vive en RAM. El feed responde en microsegundos.

Eso es persistencia políglota — no usar la misma herramienta para todo, sino elegir la correcta para cada problema.
